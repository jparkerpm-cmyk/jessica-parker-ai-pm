## What Is Here

Each project folder contains three things: the tool itself, a product decision log documenting every significant choice made during the build, and a short README with context and a live link where available.

The decision logs are the point. Anyone can ship a working HTML file. The log shows the problem definition, the constraints, the tradeoffs, and what broke in actual use. That is the work.

### Walking Pad Workout Widget
A browser-based workout tracker built for a treadmill desk. Single HTML file, no backend, timer-driven with automated phase transitions and equipment swap options.

Built to solve a specific workflow problem: eliminating pre-session decisions so the routine runs without requiring active management.

## How I Build

Every tool here started with a specific, named problem. Not a feature idea. A workflow that was broken or missing.

The build process is the same one I use with clients. Define the actual problem. Map how the user operates. Make the smallest thing that removes the friction. Observe what breaks. Iterate.

I build these tools for my own use, which means I am always the first user. That is not a limitation. It is the fastest path to an honest requirements definition.
